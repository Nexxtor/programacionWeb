VCS: Git

Guía rápida

Git.

Sistema de control de versiones.

Introducción

Git es un sistema de control de versiones que facilita la gestión de los diversos cambios que se realizan sobre los elementos de algún producto o una configuración del mismo. Una versión, revisión o edición de un producto, es el estado en el que se encuentra el mismo en un momento dado de su desarrollo o modificación.

La presente guía está destinada a facilitar el uso de la herramienta de control de versiones Git, en conjunto con el servicio en línea de Github.com; sin embargo se considera válido el contenido de la misma para otros servicios en la web del mismo estilo.

Libro de referencia de Git.

Para dominar lo temas mas avanzados que involucran a git, como el uso de ramas, se recomienda la lectura del libro de referencia, alojado en la mainPaga de git.

Referencia a Libro de referencia de Git: Reference Manual.

Contextualizacón

En primer lugar se debe crear un repositorio en cualquier servicio de control de versiones en línea. En este caso se hará uso del siguiente repositorio ficticio: https://github.com/DouglasHdezT/repositorio.git.

Además se deberá establecer un espacio de trabajo local (directorio), con el fin de el proyecto en si. En este caso será el directorio /home/douglashdezt/PW-2018.

Inicialización local y conexión remota

Inicialización local

Este paso marca la creación y configuración de un repositorio de manera local; en este punto ya existen los recursos necesario para llevar el control de un proyecto, con el único detalle que no se encuentra en línea.

Cabe resaltar que la dirección de la consola debe de estar en el directorio en el que se encuentre el proyecto;para el ejemplo, /home/douglashdezt/PW-2018. El comando a ingresar es el siguiente: 

                
    git init

Este genera los ficheros de configuración necesarios para que el sistema de control de versiones reconosca el proyecto, y pueda llavar la referencia de cada uno de los archivos dentro del mismo.

Conexión remota

Este paso consiste en indicar a Git la conexión del repositorio local con uno alojado en un sevicio en línea. Para este caso se hará uso del repositorio ficticio mencionado con anterioridad, el cual actuará como remoto del repositorio local.

Antes de ingresar el comando se debe verificar que la dirección de la consola se encuentre en la raiz del proyecto (donde se realizó git init), ya que caso contrario fallará. La sintaxis del comando para realizar la conexión es la siguiente: 

                
    git remote add <remote_name> <repository_url>

Para el caso del repositorio ficticio: 

                
    git remote add origin https://github.com/DouglasHdezT/repositorio.git

Configuración de variables de usuario.

Para llevar el control de versiones de cualquier proyecto con Git, se requiere especificar el usuario y correo electrónico que realiza los cambios. Por ello se debe modificar los valores de las variables user.name y user.email dentro de la configuración del repositorio local. Se recomienda hacer uso tanto del usuario como del correo configurados en la cuenta de GitHub.

La sintaxis de los comandos, que sirven para realizar dichos cambios, es la siguiente: 

                
    git config user.name "<username>"

    git config user.email "<email@service.ext>"

Variables Globales

El paso anterior debe de repetirse por cada repositorio local que se inicialice dentro de una computadora. En caso que la computadora en la que se trabaje posea un solo dueño o solo una persona la utilice, puede agregarse el modificador --global a los comandos anteriores.

Este modificará las variables de usuario dentro de la configuración global de Git, por lo que se evita la repetición de dichos comandos en cada proyecto. 

                    
    git config --global user.name "<username>"

    git config --global user.email "<email@service.ext>"

Resgistro de versión.

Como se mencionó con anterioridad, registrar una versión consiste guardar el estado de un producto o proyecto en cierto momento de su desarrollo. Dentro de Git existen una paleta de comandos que facilitan esta tarea.

Preparar commit.

Preparar el commit consiste en indicarle a Git que archivos o directorios comprenden la nueva versión. El comando específico para indicar este estado es add, seguido del fichero a agregar. 

                
    git add <file|directory>

Para fines prácticos, se recomienda sgregar todos los archivos posibles dentro de la versión. Para ello, en vez de agregar cada archivo uno por uno, se utiliza el "." (punto) para indicar que todos los archivos con cambios serán agregados en el commit. 

                
    git add .

Realizar el commit

Una vez se tengan tengan todos archivos preparados, se realiza la publicación local de una versión; esta va de la mano con un mensaje que explique de forma breve en que consiste la actualización. Esto significa que, por el momento, el versionamiento del código solo se encuentra dentro de la computadora donde se ha trabajado. 

                
    git commit -m "<message>"

Para el ejemplo particular: 

                
    git commit -m "Arreglando la estructura del proyecto"

A partir de este punto se pueden realizar todo tipo de acciones con las versiones publicadas, desde regresar a una en específico, hasta borrar un serie de commits poco útiles, etc.

Publicar las versiones.

El último paso consiste en "subir" las versiones publicadas localmente a un repositorio alojado en un servicio en línea, como GitHub. Para ello simplemente debe realizarce el siguiente comando, donde se especifica el remoto (Asignado previamente) y la rama que se quiere publicar (por el momento solo se ha hecho uso de la rama master). 

                
    git push <remote_name> <branch_name>

Para ejemplificar: 

                
    git push origin master

Si toda la configuración, preparación y publicación de versiones se ha realizado de forma correcta, en este punto la consola pedirá al usuario sus credenciales de GitHub. Al ingresarlas de manera conrrecta se realizará automaticamente la publicación de la versión en línea.
